# Audit of the PR #905 Arb certificate source binding

## Status

```text
AUDIT
Verdict: OPEN GAP
Finding: SHOULD — CERT BINDING/STALENESS
Target: PR #905 head 00009641, integrated at 3404d21
```

The finite Arb calculation was **not replayed**.  This is a transcript- and
SHA-256-only audit under the stated environment constraint: no `python-flint`,
Sage, Lean, or other heavy replay.  The prior #911 audit remains the authority
for the independently checked symbolic half.

## Claim

The committed `pass: true` certificate does not bind to the PR #905 files it
purports to certify.  All six SHA-256 fields generated by
`certificate_payload` disagree with the exact target bytes at the pinned PR
head; they also disagree after integration at `3404d21`.  A pinned snapshot of
the current PR body identifies the committed certificate payload, but its claim
that artifact hashes passed contradicts the committed manifest.

This is a provenance gap, not a counterexample to TS1--TS3.  The finite Arb
result may be mathematically correct, but the shipped evidence cannot identify
which proof/verifier/contract bytes produced it.

## Findings

### SHOULD — the certificate is stale against all six bound sources

The certificate records the same stale values as `SHA256SUMS.txt`.  Direct
hashing of the exact #905 tree gives:

| Certificate field | Recorded | Actual at `00009641` | Actual at `3404d21` |
|---|---:|---:|---:|
| `proof_sha256` | `0cf65b76...` | `5fc9f180...` | `5fc9f180...` |
| `source_sha256` | `1fd9d8bb...` | `9eac9446...` | `9eac9446...` |
| `replay_source_sha256` | `47e818c4...` | `32b45f42...` | `32b45f42...` |
| `consumer_contract_sha256` | `3a3f1558...` | `a0cfa97b...` | `a0cfa97b...` |
| `consumed_note_sha256` | `6ebb6f6e...` | `e92c61aa...` | `e13be4f1...` |
| `consumer_script_sha256` | `5fbe9ef7...` | `561bcf5d...` | `23703024...` |

Thus all `6/6` bindings mismatch (`0/6` match) at both the producer head and
integration point.
This is stronger than the already-recorded `10/10` manifest mismatch: the
certificate payload itself contains stale provenance.

The verifier recomputes exactly these six fields before serializing its
payload.  Its `--check` branch requires byte-for-byte equality with the
committed JSON.  Therefore a run of the committed verifier that reaches the
comparison cannot report `certificate_check: PASS` against the committed
certificate, regardless of the numerical gate results.

The certificate pins the upstream base `c4856fa`, not its producing head, and
no hash-bound stdout or tamper transcript is checked in.  The historical source
bundle matching the six recorded hashes is unavailable in the PR history, which
contains one commit.

### SHOULD — the pinned PR-body replay summary is not coherent with the tree

The PR-body snapshot at `updatedAt=2026-07-18T10:13:57Z` has CLI body-output
SHA-256 (raw Markdown plus the trailing LF emitted by `gh --jq`)
`40be2b98b36423d5037871e4b915f3d746fae78b41e8544d87a728951890dce9`.
It prints certificate SHA-256
`80c8dfc072977d4ea6359086f7dedf4eaa67a5a56b6064634299a933f02d4e09`.
That equals the committed certificate file, so the payload identity is clear.
The same summary says `artifact hashes PASS`, while every one of the ten
committed manifest entries mismatches at the same pinned head.  Consequently
the summary cannot be the output of the committed replay wrapper on the
committed tree.

The wrapper order is also important for containment: lines 39--45 run Arb
`--check`, Arb tamper tests, compilation, and the class-charge deep verifier
*before* `replay_hashes()`.  The prior #911 audit correctly found the stale
manifest but incorrectly said the manifest abort occurs before `python-flint`.
It also concluded that only the manifest was stale; the `6/6` internal binding
mismatch above supersedes that narrower packaging diagnosis.

### NOTE — current integration contains the consequence

The repaired class-charge ledger at `3404d21`, section 3.5 lines 409--429,
already carries the all-depth transfer as `INV-TAIL`, explicitly open modulo
#905's finite Arb certificate.  Its P15 check binds the class-charge
certificate to its own note and script; it does not validate #905's certificate
or restore its missing provenance.  No current paper theorem is promoted by
this audit.

## Verification

The companion verifier is standard-library-only.  Its default mode is
self-contained and reads no repository file: it embeds the pinned full hashes,
checks the `10/10` manifest and `6/6` internal-binding failures at both target
points, identifies the certificate payload, records the wrapper order, and
exercises five semantic tamper mutations.  The optional `--verify-tree` mode
re-parses both Git objects, reloads the manifest/certificate, hashes all target
blobs, derives wrapper order from source, and checks every embedded observation.

```bash
python3 experimental/scripts/verify_dense_shell_transfer_shape_cert_binding_audit.py
python3 experimental/scripts/verify_dense_shell_transfer_shape_cert_binding_audit.py --tamper-selftest
python3 experimental/scripts/verify_dense_shell_transfer_shape_cert_binding_audit.py --verify-tree
python3 experimental/scripts/verify_dense_shell_transfer_shape_audit.py
python3 experimental/scripts/verify_dense_shell_class_charges.py
```

Expected summaries:

expected to fail all entries; that failure is the audited observation, not a
green verification command.

## Consequence and closure condition

Keep the finite Arb leg `OPEN GAP` and do not use the committed `pass: true`
JSON as source-bound evidence for `PROVED` until one of these is supplied:

1. the exact historical source bundle matching all six embedded hashes,
   together with a hash-bound successful transcript **and** an audited
   statement/contract bridge proving that the historical obligations imply the
   current `00009641` / `3404d21` obligations (otherwise it certifies only the
   historical bundle); or
2. a capable environment re-emits the certificate and manifest from the exact
   committed bytes, runs `--check` and the tamper suite, and checks in or
   otherwise immutably binds the complete transcript.

No lower reserve, product-profile admission, general decorated-subtree
positivity, or complete asymptotic theorem follows from this packet.
